# 학습노트 — 코딩 세션에서 나온 질문 정리

> 개발 진행하면서 궁금했던 것들을 나중에 다시 볼 수 있게 정리한 파일. 실행 계획 문서 아님, 개인 학습용.

---

## 1. "가상환경"은 왜 파이썬 venv가 아니었나

**질문**: 가상환경 만들자고 했는데 왜 Node.js 얘기가 나왔나?

**답**: "가상환경(virtual environment)"은 파이썬 생태계 용어(`venv`, `conda` 등 — 프로젝트마다 라이브러리 버전이 안 섞이게 격리하는 것)다. 이 프로젝트는 Next.js/TypeScript(자바스크립트 생태계)라서 같은 역할을 하는 게 **Node.js 프로젝트 하나(`package.json` + `node_modules`)** 다. 언어 생태계마다 "격리된 개발 환경"을 부르는 말이 다르다:

| 생태계 | 격리 단위 |
|---|---|
| Python | venv, conda |
| Node.js/JS | package.json + node_modules (프로젝트 폴더 자체가 격리 단위) |

## 2. Node.js가 컴�터에 아예 없었던 이유와 설치 방법

이 컴퓨터엔 Node.js가 설치된 적이 없었다. `nvm-windows`(Node 버전 관리자)를 깔고, 그걸로 Node LTS(24.18.0)를 설치했다.

**왜 nvm을 거쳐서 까나?** 나중에 프로젝트마다 다른 Node 버전이 필요해질 수 있어서 (예: 이 프로젝트는 24, 다른 프로젝트는 20) 버전을 자유롭게 바꿔 쓸 수 있게 하는 게 nvm의 역할이다. 파이썬의 `pyenv`와 같은 개념.

**설치 직후 왜 바로 안 됐나?** Windows는 환경변수(PATH)를 레지스트리에 저장하는데, **이미 열려 있는 터미널/프로세스는 이 변경을 자동으로 못 받는다.** 새 터미널을 열어야 반영된다. (Claude Code 도구가 쓰는 셸은 세션 내내 재사용되는 구조라 계속 직접 경로를 지정해야 했다.)

## 3. Next.js 15 vs 16, Tailwind v3 vs v4 — 왜 문서랑 실제가 달랐나

CLAUDE.md는 "Next.js 15"라고 적어뒀는데, 지금 `create-next-app`으로 설치하니 **Next.js 16**이 깔렸다. 문서를 쓴 시점 이후 새 버전이 나온 것뿐이다. **패키지 버전은 시간이 지나면 문서보다 항상 앞서 나간다** — 새 프로젝트를 시작할 때는 문서에 적힌 옛날 버전을 억지로 맞추기보다, 그 시점 최신 안정판으로 시작하고 문서를 그에 맞게 갱신하는 게 낫다 (지금 그렇게 했다).

**Tailwind v3 → v4의 진짜 변화**: v3까지는 `tailwind.config.js`라는 JS 파일 안에 색상·폰트 같은 "테마"를 객체로 적었다. v4부터는 이 방식을 버리고, CSS 파일(`globals.css`) 안에 `@theme { --color-ink: #12253F; }` 처럼 **CSS 변수로 직접** 적는다. JS 설정 파일 자체가 필요 없어졌다 — "설정을 CSS 안으로 가져온다"는 큰 방향 전환이다.

## 4. 백엔드는 파이썬이 아니다 — 왜 파이썬을 쓰고 있었나

이 프로젝트의 실제 백엔드는 **없다(Phase 1)** → 나중에 앱 만들 때도 **Node.js + PostgreSQL**로 이미 정해져 있음(파이썬 아님).

그런데 실측 데이터(경로·고도)를 모으는 동안 파이썬을 썼던 이유: `scripts/build-profiles.ts`처럼 **한 번만 실행하고 끝나는 배치 스크립트**는 앱에 포함되지 않으니 어떤 언어로 짜든 상관없다. 파이썬이 API 호출 + 데이터 가공을 빠르게 실험하기 편해서 프로토타입으로 먼저 썼고, 검증되면 최종본은 TypeScript로 옮겨 적는다.

**핵심 구분**: "앱이 실행될 때 도는 코드"(TypeScript만) vs "데이터를 미리 만들어두는 일회성 도구"(아무 언어나 가능, 결과물인 데이터 파일만 앱에 들어감).

## 5. OpenStreetMap(OSM) 데이터를 실제로 어떻게 가져왔나

- **Overpass API**: OSM 데이터베이스에 질의(쿼리)를 던져서 원하는 지도 데이터만 뽑아오는 서비스. "Overpass QL"이라는 전용 질의어를 쓴다.
- **relation(관계)**: OSM에서 "이 길들을 순서대로 이으면 하나의 도보길이 된다"처럼 여러 요소를 묶는 단위. 카미노 프랑스 길은 **relation 하나가 아니라 6개 구간(relation)으로 쪼개져** 있었다 (생장→로그로뇨, 로그로뇨→부르고스, ...).
- **way(선)**: relation 안에 들어있는 실제 도로/오솔길 조각. 한 구간(relation)은 수백 개의 way로 이루어져 있고, **순서가 뒤섞여** 온다 — 그래서 "끝점이 서로 맞는 조각을 이어붙이는" 스티칭(stitching) 작업이 필요했다.
- **검증**: 이렇게 이어붙인 경로 길이가 163.3km 나왔는데, 원래 알고 있던 값(162.2km)과 거의 일치해서 "제대로 이었다"는 걸 확인할 수 있었다.

## 6. 고도 데이터는 왜 "IGN 5m 격자"가 아니라 "EU-DEM 25m"을 썼나

CLAUDE.md는 스페인 국립지리원(IGN)의 MDT05(5m 격자, 가장 정밀함)를 쓰라고 돼 있다. 그런데 IGN 데이터는 **공개 API가 아니라 직접 파일(GeoTIFF)을 다운로드해서 좌표계 변환까지 해야 하는** 훨씬 무거운 작업이다.

그 대신 **Open Topo Data**라는 무료 공개 API가 있고, 여기서 "EU-DEM 25m"(유럽 전역 25m 격자, CC-BY 라이선스)을 바로 조회할 수 있어서 먼저 이걸로 파이프라인 전체를 검증했다. **5m보다 거칠지만 "마을 고도차로 계산"하는 것보다는 압도적으로 정확하다.**

**검증 결과**: 생장→론세스바예스 구간 실제 상승고도를 1,327m로 계산했는데, CLAUDE.md에 이미 적혀있던 기준값(약 1,280m)과 4% 차이. 오차 원인 후보: (1) 25m 격자라 5m보다 거칢 (2) 스무딩(평활화) 방식 차이 (3) OSM에 그려진 실제 경로와 가이드북 경로가 살짝 다를 수 있음.

## 7. "노이즈 때문에 스무딩이 필수"라는 게 무슨 뜻인가

고도 데이터를 아무 가공 없이 이어 붙이면(raw), 지형 자체의 미세한 요철 + 측정 오차까지 전부 "오르막"으로 세어버려서 **실제보다 훨씬 크게 나온다.** (이번 검증에서 raw 1,513m vs 스무딩 후 1,327m — 스무딩 없이 그냥 더하면 14% 더 크게 나왔다.)

**스무딩 방법 두 가지를 같이 씀**:
1. **이동평균(moving average)**: 앞뒤 몇 개 지점의 고도를 평균 내서 뾰족한 튐을 눌러준다.
2. **히스테리시스(hysteresis) 임계값**: "3m 이상 방향이 바뀌어야 그제서야 오르막/내리막으로 인정"하는 규칙. 등산 앱들이 실제로 쓰는 표준적인 방식이다.
