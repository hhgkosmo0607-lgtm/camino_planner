# 1부 · Claude Code 실전 사용법

---

## 0. 핵심 원칙 세 가지

1. **컨텍스트를 파일로 관리한다.** 매번 프롬프트에 붙여넣지 않는다
2. **큰 작업은 계획을 먼저 받는다.** 바로 코드를 쓰게 하지 않는다
3. **되돌릴 수 있게 해둔다.** 커밋과 체크포인트가 안전망이다

> Claude Code는 기능이 자주 바뀐다. 이 문서와 실제 동작이 다르면 `/help`와 공식 문서(https://code.claude.com/docs)를 따른다.

## 1. 최초 설정 (한 번만)

### 1.1 저장소 준비

```bash
mkdir camino && cd camino
git init
mkdir docs
# 01_조사와근거.md, 03_제품과기능.md, 03_제품과기능.md 를 docs/ 에 복사
# CLAUDE.md 를 루트에 복사
# camino-companion.jsx 를 루트에 복사 (P0에서 데이터 이관용)
git add -A && git commit -m "문서와 프로토타입 추가"
```

**`CLAUDE.md`는 루트에 둔다.** 매 세션 자동으로 읽힌다.

### 1.2 Claude Code 실행

```bash
claude
```

### 1.3 첫 세션에서 할 일

```
/init          # 저장소를 스캔해 CLAUDE.md 초안 생성 (이미 있으면 병합 검토)
/memory        # CLAUDE.md 내용 확인·수정
/permissions   # 승인 규칙 설정
```

우리는 이미 CLAUDE.md를 작성했으므로 `/init`은 선택이다. 다만 `/init`이 프로젝트 구조를 스캔해 빌드·테스트 명령을 찾아주므로, **실행한 뒤 우리 내용과 병합**하는 게 낫다.

### 1.4 CLAUDE.md에 대한 주의

- **틀린 내용을 쓰면 없는 것보다 나쁘다.** Claude가 자신 있게 반복한다
- 매 세션 컨텍스트를 차지하므로 **길수록 손해다.** 오래된 문단은 지운다
- 규칙은 **동작으로** 쓴다. "코드를 깔끔하게" ❌ / "계산 로직은 lib/planner에 순수 함수로" ✅
- 개인 설정은 `CLAUDE.local.md`에 (gitignore 대상)

## 2. 프롬프트를 먹이는 방법

### 2.1 기본 흐름 — 한 번에 하나

```
1. /clear           ← 이전 작업 컨텍스트 비우기 (CLAUDE.md는 유지됨)
2. P0 프롬프트 붙여넣기
3. 결과 확인 → 완료 조건 직접 검증
4. git commit
5. 1번으로 돌아가 P1
```

**P0~P7을 한 번에 주지 않는다.** 여러 작업을 섞으면 각각의 품질이 떨어지고, 어디서 잘못됐는지 추적이 안 된다.

### 2.2 큰 작업은 계획부터

P1(일정 엔진)이나 P5(SEO 페이지)처럼 파일을 여러 개 건드리는 작업은 **먼저 계획을 받는다.**

```
/plan docs/06_개발가이드.md 의 P1 을 구현할 계획을 세워줘.
     파일을 아직 수정하지 말고 계획만.
```

`/plan`은 **읽기 전용 모드**다. 코드를 분석하고 계획서를 쓰지만 파일을 수정하거나 명령을 실행하지 않는다. 계획을 읽고 마음에 들면 승인한다.

**계획을 읽을 때 확인할 것**
- 우리가 지정한 타입 이름을 그대로 쓰는가
- 테스트를 먼저 만드는가, 나중에 만드는가 (먼저가 낫다)
- 불필요한 의존성을 추가하려 하는가
- CLAUDE.md의 금지 사항을 어기는 부분이 있는가

### 2.3 프롬프트를 파일로 참조하기

긴 프롬프트를 매번 붙여넣는 대신 이렇게 해도 된다.

```
docs/06_개발가이드.md 의 P1 섹션을 읽고 그대로 구현해줘.
완료 조건까지 전부 만족시켜야 해.
```

**장점**: 프롬프트 원문이 저장소에 남아 재현 가능하다.
**단점**: 파일을 읽는 데 컨텍스트를 쓴다.

P0~P2처럼 타입 정의가 중요한 단계는 **직접 붙여넣는 쪽**을 권한다. 정확도가 중요하다.

### 2.4 반복 작업은 스킬로 만든다

같은 지시를 자주 한다면 커스텀 커맨드(스킬)로 만든다. `.claude/skills/` 아래에 `SKILL.md`로 저장하면 `/이름`으로 호출된다.

예: `.claude/skills/verify/SKILL.md`

```markdown
---
name: verify
description: 변경 후 검증 루틴
---

아래를 순서대로 실행하고 결과를 보고해줘.

1. npm test
2. npm run build
3. lib/planner/ 안에 fetch, window, localStorage 참조가 있는지 grep
4. data/towns.ts 의 마을 수와 km 정렬 상태 확인

하나라도 실패하면 원인을 설명하고, 고치기 전에 먼저 물어봐.
```

이러면 매번 `/verify` 한 줄로 끝난다.

## 3. 세션 관리

### 3.1 언제 `/clear` 하나

**작업 단위가 바뀔 때마다.** P0이 끝나고 P1로 갈 때가 그 시점이다.

`/clear`는 대화를 비우지만 **CLAUDE.md 같은 프로젝트 메모리는 유지**한다. 이전 작업의 시행착오가 다음 작업의 판단을 흐리는 걸 막는다.

### 3.2 언제 `/compact` 하나

**한 작업 안에서 대화가 길어졌을 때.** 요약해서 컨텍스트를 비우되 대화는 이어간다.

```
/context     # 지금 뭐가 컨텍스트를 차지하는지 확인
/compact     # 요약해서 공간 확보
```

큰 작업을 시작하기 전에 `/compact`를 한 번 돌리면 안정적이다.

### 3.3 위험한 변경 전에는 `/branch`

되돌릴 자신이 없는 실험은 대화를 분기해두고 한다. 원래 대화는 `/resume`으로 돌아갈 수 있다.

## 4. 검증 — 이게 제일 중요하다

### 4.1 완료 조건을 직접 확인한다

프롬프트마다 완료 조건을 숫자로 써둔 이유가 이것이다. **Claude가 "완료했습니다"라고 해도 직접 본다.**

```bash
npm test              # P1 이후 필수
npm run build         # P5 이후 필수, 정적 생성 페이지 수 확인
npm run dev           # 눈으로 확인
```

P5(SEO)는 특히 이렇게 확인한다.

```bash
npm run build 2>&1 | grep -c "○\|●"   # 생성된 페이지 수
curl -s localhost:3000/town/sarria | grep -c "<h1"  # 서버 렌더 확인
```

### 4.2 변경 내용을 본다

```
/diff              # 대화형 diff 뷰어
/code-review       # 현재 diff의 버그·정리 대상 검토
```

`/code-review --fix`로 발견 사항을 바로 적용할 수도 있다. 다만 **자동 수정 후에도 `/diff`로 다시 본다.**

### 4.3 이 프로젝트에서 특히 봐야 할 것

| 확인 | 이유 |
|---|---|
| `data/towns.ts`의 km·elevation이 안 바뀌었나 | 검증된 데이터다. 임의 수정 금지 |
| 알베르게 데이터를 지어내지 않았나 | 사람이 한데서 잔다 |
| 원어명(`nameEs`)이 UI에서 안 사라졌나 | 현지에서 보여줘야 한다 |
| `lib/planner/`에 fetch가 들어갔나 | 오프라인 동작이 깨진다 |
| 페이지가 서버 렌더되나 (JS 끄고 확인) | SEO가 전략 그 자체다 |
| 글꼴이 17px 아래로 내려갔나 | 60대 사용자 |

**`git diff data/`를 습관적으로 확인한다.** 데이터 파일 변경은 거의 항상 실수다.

## 5. 잘못됐을 때

### 5.1 되돌리기

```
/rewind      # 코드와 대화를 체크포인트로 되돌린다
```

git 커밋이 있으면 더 확실하다.

```bash
git checkout -- .        # 커밋 안 한 변경 버리기
git reset --hard HEAD    # 마지막 커밋으로
```

**각 프롬프트 완료 후 반드시 커밋한다.** 이게 최선의 안전망이다.

### 5.2 자주 나는 문제와 대응

| 증상 | 원인 | 대응 |
|---|---|---|
| 없는 알베르게를 만들어냄 | 빈 필드를 채우려는 습성 | CLAUDE.md 규칙 1 재확인 시키고 `/rewind` |
| 타입 이름을 자기 식으로 바꿈 | 프롬프트에 타입을 안 줬거나 무시 | 타입 정의를 다시 주고 재작업 |
| 테스트를 자기 코드에 맞춰 씀 | 테스트를 나중에 시켰음 | 프롬프트의 테스트 케이스를 명시적으로 지정 |
| `useEffect`에서 계산 | 습관적 클라이언트 렌더 | "서버 컴포넌트에서 계산" 재지시 |
| 의존성을 마음대로 추가 | | CLAUDE.md에 "추가 전 이유 설명" 규칙 있음. 롤백 |
| 컨텍스트가 꽉 참 | 대화가 길어짐 | `/context` → `/compact` 또는 `/clear` |

### 5.3 같은 실수가 반복되면

**CLAUDE.md에 규칙을 추가한다.** 매번 지적하는 건 낭비다.

```
/memory
```

로 열어서 "하지 말 것"에 한 줄 추가한다.

## 6. 이 프로젝트의 권장 순서

```
[준비]
  저장소 생성, 문서 배치, CLAUDE.md 루트에
  claude 실행 → /init → /permissions

[P0] 프로젝트 초기화 + 데이터 이관
  ⚠️ 착수 전: 01 문서 2부 결함 1·2 해결 여부 확인
     (GPX에서 ascent/descent 추출 계획)
  → 붙여넣기 → 확인 → git commit
  /clear

[P1] 일정 분할 엔진      ← 가장 중요. /plan 먼저
  → /plan 으로 계획 받기 → 승인 → 구현
  → npm test 전부 통과 확인
  → git commit
  /clear

[P2] 시그니처 컴포넌트
  → 붙여넣기 → npm run dev 로 눈 확인 → commit
  /clear

[P3] 계산기 화면
  → JS 끄고 서버 렌더 확인 필수
  → Lighthouse 실행
  → commit
  /clear

[P4] 인쇄용 일정표
  → 실제로 인쇄 미리보기 확인
  → commit
  /clear

[P5] SEO 페이지          ← /plan 먼저
  → npm run build 로 페이지 수 확인
  → commit
  /clear

[P6] 무료 도구 3종
  → 각각 독립 확인
  → commit
  /clear

[P7] 계측 (공개 직전)
  → commit → 배포
```

**P1과 P5는 `/plan`을 먼저 쓴다.** 파일을 여러 개 건드리고 되돌리기 번거로운 작업이다.

## 7. 프롬프트 작성 요령 (직접 추가할 때)

05번 문서의 프롬프트를 수정하거나 새로 쓸 때 참고.

### 7.1 좋은 프롬프트의 구조

```
[무엇을] 한 문장으로
[어디에] 파일 경로를 명시
[타입]   인터페이스를 직접 제공
[규칙]   알고리즘을 단계로
[검증]   테스트 케이스를 미리 나열
[완료]   숫자로 측정 가능하게
[금지]   하지 말 것을 명시
```

### 7.2 반드시 지킬 것

| 원칙 | 이유 |
|---|---|
| **타입 정의를 직접 준다** | 이름이 흔들리면 뒤 단계가 전부 어긋난다 |
| **테스트 케이스를 미리 쓴다** | 나중에 시키면 자기 코드에 맞춘 테스트를 쓴다 |
| **완료 조건을 숫자로** | "잘 동작한다"는 검증 불가 |
| **하지 말 것을 명시** | 안 시킨 일을 하는 쪽으로 기운다 |
| **한 프롬프트 = 한 관심사** | 섞으면 품질이 떨어진다 |
| **도메인 함정을 미리 알린다** | "공립 알베르게는 예약을 안 받는다" |
| **★로 우선순위 표시** | 전부 중요하면 아무것도 안 중요하다 |

### 7.3 나쁜 예 / 좋은 예

**나쁜 예**
```
일정 계산 기능이랑 SEO 페이지 만들어줘. 예쁘게 해줘.
```
- 두 관심사가 섞임
- "예쁘게"는 검증 불가
- 타입, 완료 조건, 금지 사항 없음

**좋은 예**
```
lib/planner/split.ts 에 구간 분할 함수를 만들어줘.
순수 함수로. fetch·window·localStorage 사용 금지.

[타입 정의 제공]

알고리즘:
1. 체력 보정 (low ×0.85 / normal ×1.0 / high ×1.15)
2. ...

테스트 (반드시 작성):
- 사리아 출발 → 총 114.2km
- 목표 14km에서 무한루프 없음
- ...

완료 조건: npm test 전부 통과, split.ts에 fetch 참조 없음

하지 마: 마을 데이터의 km 값을 수정하지 마
```

## 8. 비용 관리

```
/usage       # 토큰 사용량 확인 (별칭 /cost)
/context     # 무엇이 컨텍스트를 차지하는지
/effort      # 추론 수준 조정
```

**절약 요령**

- 단순 작업(P2 컴포넌트, P4 인쇄)은 효율을 낮춰도 된다
- 복잡한 작업(P1 엔진, P5 SEO)은 높인다
- CLAUDE.md를 짧게 유지한다. 매 세션 컨텍스트를 먹는다
- 작업 단위마다 `/clear`. 대화가 길어질수록 매 턴 비용이 오른다
- 큰 파일을 통째로 읽히지 말고 경로와 범위를 지정한다

## 9. 주의사항

### 9.1 자동 승인을 함부로 켜지 않는다

`/permissions`에서 승인 규칙을 설정할 수 있지만, **파일 삭제나 `git reset` 같은 명령까지 자동 승인하면 위험하다.** 읽기 전용 명령 정도만 미리 허용한다.

`/fewer-permission-prompts`가 자주 쓰는 읽기 전용 명령을 찾아 허용 목록을 만들어준다.

### 9.2 이 프로젝트 특유의 위험

**Claude Code는 빈 필드를 채우려는 습성이 있다.** 알베르게 데이터에서 이게 발동하면 없는 숙소가 생긴다.

`CLAUDE.md`의 규칙 1을 최우선으로 둔 이유가 이것이고, **알베르게 데이터는 실측 도보 전까지 아예 만들지 않는 것**이 가장 안전하다.

### 9.3 문서와 코드가 어긋나면

01 문서 2부의 결함 1·2처럼 **사양 자체가 틀린 경우가 있다.** Claude Code는 사양대로 정확히 구현하므로, 틀린 사양은 틀린 코드가 된다.

**의심스러우면 구현 전에 물어본다.**

```
03_제품과기능.md 의 F-01 알고리즘을 읽고,
논리적 결함이나 구현이 불가능한 부분이 있는지 먼저 검토해줘.
코드는 아직 쓰지 마.
```

## P0.5 — 경로·고도 프로파일 생성 🔴

> **03 문서 부록을 먼저 읽는다.** 라이선스와 평활화 함정이 정리돼 있다.
> 이 단계 없이 P1을 하면 부상 회피 엔진이 최대 39% 틀린 값으로 동작한다.

```
scripts/build-profiles.ts 를 만들어줘.
일회성 배치 스크립트다. 서비스 런타임 코드가 아니다.

목적: 마을 82곳 사이 81개 구간의 실제 누적 상승/하강을 계산한다.

입력
1) 경로 좌표 — OpenStreetMap 카미노 프랑스 길 릴레이션
   Overpass API로 조회. 결과를 data/geometry/frances.geojson 에 저장.
   ★ 이 파일은 .gitignore 에 넣어줘 (ODbL 파생 DB 배포 회피)

2) 고도 — 스페인 IGN MDT05 (COG GeoTIFF, 5m 격자)
   centrodedescargas.cnig.es 에서 받은 파일을 data/dem/ 에 둔다고 가정.
   파일이 없으면 스크립트는 안내 메시지를 출력하고 종료.
   ★ DEM 파일을 자동 다운로드하지 마. 도엽 선택은 사람이 한다

처리
1) 경로를 마을 좌표 기준으로 81개 구간으로 분할
2) 각 구간을 15m 간격으로 리샘플링
3) 각 점의 고도를 DEM에서 이중선형 보간으로 샘플링
4) ★ 평활화: 이동평균(윈도우 5) 적용 후,
   연속 고도차가 4m 미만이면 누적에서 제외
5) ascent = Σ max(0, Δh), descent = Σ max(0, -Δh)
6) maxElevation = 구간 최고점

출력
data/profiles.ts — SegmentProfile[] , source: 'OSM+MDT'
★ 좌표는 넣지 마. 숫자만 남긴다

검증 (스크립트가 자체 실행)
계산 후 아래를 콘솔에 출력하고, 벗어나면 경고를 띄워줘.
- 생장 → 론세스바예스: 누적 상승 1,200~1,400m 기대
- 비야프랑카 → 오 세브레이로: 700~900m 기대
- 엘 아세보 → 몰리나세카: 하강 500m 이상, 상승은 거의 0 기대
- 부르고스 → 오르니요스 (메세타): ★ 상승 100m 이하여야 정상
  이 값이 300m를 넘으면 평활화가 안 된 것이다. 크게 경고해줘

완료 조건:
- npm run build:profiles 로 실행된다
- data/profiles.ts 에 81개 레코드, 전부 source:'OSM+MDT'
- 위 검증 4개가 전부 기대 범위 안에 들어온다
- data/geometry/ 가 .gitignore 에 있다
- README 에 출처 표시 문구가 추가돼 있다:
  "경로 데이터 © OpenStreetMap contributors (ODbL)"
  "고도 데이터 © Instituto Geográfico Nacional (CC BY 4.0)"

하지 마:
- ★ 계산이 안 되면 그럴듯한 값을 지어내지 마. 실패로 종료해
- DEM 파일을 자동으로 받지 마
- 좌표를 profiles.ts 에 넣지 마
- 이 스크립트를 서비스 코드에서 import 하지 마
```

## P1 — 부상 회피형 일정 분할 엔진

```
lib/planner/ 에 구간 분할 엔진을 구현해줘.
순수 함수로만 작성하고 React/Next.js에 의존하지 마.
나중에 React Native로 그대로 옮길 코드야. fetch, window, localStorage 사용 금지.

파일: lib/planner/{types.ts, split.ts, risk.ts, split.test.ts}

★ 이 엔진의 목표는 "최단 일수"가 아니라 "부상 없는 완주 확률"이다.
  경쟁 앱은 사용자가 직접 구간을 고르게 두고 무리한 일정을 방치한다.
  우리는 무리한 일정을 능동적으로 교정한다.

타입:

export interface PlanInput {
  startTownId: string
  targetKmPerDay?: number      // 14~32
  totalDays?: number           // targetKm 대신 일수 지정 가능
  fitness: 'low' | 'normal' | 'high'
  restDays: number             // 기본 1(전구간) / 0(100km)
}

export type StageWarning =
  | 'STEEP_CLIMB'       // 누적 상승 600m 초과
  | 'STEEP_DESCENT'     // 누적 하강 500m 초과 ★ 무릎 부상 주원인
  | 'LONG_DISTANCE'     // 28km 초과
  | 'FEW_BEDS'          // 도착지 침대 30개 미만
  | 'NO_SERVICES'       // 물·식당 없는 구간 10km 초과
  | 'EARLY_OVERLOAD'    // 1~3일차 22km 초과
  | 'CONSECUTIVE_HARD'  // 연속 3일 26km 초과

export interface Stage {
  dayNo: number
  fromTownId: string
  toTownId: string
  distanceKm: number      // 소수점 1자리
  ascent: number          // profiles.ts에서 읽는다
  descent: number         // profiles.ts에서 읽는다
  estimatedMinutes: number
  warnings: StageWarning[]
  isRestDay: boolean
}

export interface Plan {
  stages: Stage[]
  totalKm: number
  totalDays: number
  compostelaEligible: boolean
  doubleStampRequired: boolean
  injuryRiskScore: number   // 0~100, 낮을수록 안전
  riskDataQuality: 'OSM+MDT' | 'ESTIMATED'  // ★ ESTIMATED면 UI에 점수를 숨긴다
  advice: string            // 한 줄 판단
}

★ 고도는 반드시 data/profiles.ts 의 SegmentProfile 에서 읽어라.
  towns.elevation 을 직접 빼서 오르막을 구하지 마.
  현재 profiles 는 전부 source:'ESTIMATED' 이고 실제와 최대 39% 차이가 난다.
  GPX 확보 후 교체할 것이므로 경계를 유지해야 한다.

★ 내리막이 무릎 부상의 주원인이다. 위험 산정에서 descent 가중치를
  ascent보다 높게 잡아라 (예: ascent×1.0, descent×1.4).
  단, 이 계수는 의학적 근거가 없는 임시값이므로 risk.ts 상단에 주석으로 명시해라.

알고리즘 (03_제품과기능.md F-01 참조):

1. 목표 거리 산정
   - totalDays 입력 시: daily = 총거리 / (totalDays - restDays)
   - 체력 보정: low ×0.85 / normal ×1.0 / high ×1.15

2. 탐욕 분할
   현재 위치 + 목표거리 지점에 가장 가까운 beds > 0 마을을 도착지로.
   산티아고까지 반복.

3. 부상 회피 후처리  ← 핵심
   a. 초반 적응: 1~3일차 목표거리 ×0.8
   b. 오르막 보정: 다음 구간 누적 상승 600m 초과 시 해당 구간 ×0.85
   c. 회복 배치: 상승 600m 또는 하강 500m 초과 구간의 다음 날 ×0.85
   d. 휴식일: restDays > 0 이면 beds >= 150 인 큰 도시에 배치
   e. 마지막 구간 < 6km → 직전 구간에 병합

4. 경고 산출 (위 StageWarning 기준)

5. injuryRiskScore (risk.ts)
   평균거리, 최대거리, 누적 상승, 누적 하강(가중), 초반부하, 연속고강도를 합산.
   riskDataQuality 는 구간 profiles 의 source 가 하나라도 ESTIMATED 면 'ESTIMATED'.

6. estimatedMinutes = (거리 / 4.2) * 60 + ascent / 8 + descent / 20

테스트 (반드시 작성):
- 생장 출발 24km/normal → 33~36 구간
- 사리아 출발 → 총 114.2km, compostelaEligible true, doubleStampRequired true
- 모든 구간의 fromTownId가 이전 구간의 toTownId와 일치
- 모든 도착지의 beds > 0 (산티아고 제외)
- 마지막 toTownId는 항상 santiago-de-compostela
- 거리 합계가 총 거리와 오차 0.1km 이내
- 목표 14km(최소)에서 무한루프 없음
- 목표 32km(최대)에서도 모든 도착지에 숙소 있음
- ★ 초반 적응 검증: 1~3일차 평균 거리 < 4~6일차 평균 거리
- ★ 오르막 회복 검증: 피레네 구간(생장→론세스바예스) 다음 날이 그 다음 날보다 짧다
- ★ 하강 경고 검증: 엘 아세보 → 몰리나세카가 포함된 구간에 STEEP_DESCENT 경고가 있다
  (1145m → 595m, 급경사 돌길로 무릎 부상이 많은 구간)
- ★ restDays=2 지정 시 isRestDay가 정확히 2개이고 전부 beds>=150 마을
- ★ 32km/high 전구간 → injuryRiskScore >= 60
- ★ 현재 profiles가 전부 ESTIMATED이므로 riskDataQuality === 'ESTIMATED'
- ★ split.ts / risk.ts 어디에도 towns.elevation을 직접 빼는 코드가 없다

완료 조건:
- npm test 전부 통과
- split.ts, risk.ts에 fetch/window/localStorage 참조 없음
- 고도를 profiles.ts 경유로만 읽는다
- risk.ts 상단에 "이 가중치는 의학적 근거가 없는 임시값" 주석이 있다
```

## 프롬프트 작성 원칙

이 문서를 직접 수정할 때 참고.

1. **한 프롬프트 = 한 관심사.** 데이터, 로직, UI, SEO를 섞지 않는다
2. **완료 조건을 숫자로.** "잘 동작한다"가 아니라 "82개 마을, km 오름차순"
3. **하지 말 것을 명시.** 에이전트는 안 시킨 일을 하는 쪽으로 기운다
4. **테스트 케이스를 프롬프트에 넣는다.** 나중에 시키면 자기 코드에 맞춘 테스트를 쓴다
5. **도메인 함정을 미리 알려준다.** "공립 알베르게는 예약을 안 받는다" 같은 것
6. **타입 정의를 직접 준다.** 이름이 흔들리면 뒤 단계가 전부 어긋난다
7. **★ 표시로 우선순위를 준다.** 전부 중요하다고 하면 아무것도 안 중요해진다
