# 길동무 — 개발 프롬프트 모음

> 코딩 에이전트(Claude Code 등)에 순서대로 입력한다.
> 각 프롬프트는 **독립 실행 가능**하며, 완료 조건(DoD)을 만족한 뒤 다음으로 넘어간다.

---

## 사용법

1. 저장소 루트에 `CLAUDE.md`를 넣는다 (별도 파일 참조)
2. `01_서비스기획서.md`, `02_기능사양서.md`를 `/docs`에 넣는다
3. 아래 프롬프트를 **하나씩** 붙여넣는다. 여러 개를 한 번에 주면 품질이 떨어진다
4. 각 단계가 끝나면 직접 실행해보고, 다음으로 넘어간다

**중요**: PROMPT 0을 먼저 실행해 데이터 기반을 만들어야 나머지가 작동한다.

## PROMPT 0 — 프로젝트 초기화 및 데이터 이관

```

프로젝트명은 gildongmu.

작업 순서:

   - App Router, TypeScript, Tailwind, ESLint 사용
   - src 디렉터리 사용 안 함 (app/, components/, lib/, data/ 를 루트에)
   - Vitest 설치 및 설정

2. Tailwind 토큰 등록
   CLAUDE.md의 디자인 토큰을 tailwind.config.ts의 theme.extend.colors에 등록해줘.
   이름은 ink, ink2, granite, stone, flecha, vino, moss로.
   폰트도 fontFamily에 등록: display(Gowun Batang), body(IBM Plex Sans KR),
   mono(IBM Plex Mono). app/layout.tsx에서 next/font/google로 로드.

3. 데이터 이관
   첨부한 camino-companion.jsx 파일 안의 TOWNS 배열을 data/towns.ts로 옮겨줘.
   타입을 아래와 같이 정의하고, 기존 값은 절대 수정하지 마.

   export type Service = 'WATER'|'BAR'|'SHOP'|'PHARMACY'|'ATM'|'BAG_TRANSFER'
   export interface Town {
     id: string          // 슬러그. 예: 'sarria', 'puente-la-reina'
     nameEs: string      // 원어명 (필수)
     nameKo: string
     km: number          // 생장 기준 누적거리
     elevation: number
     beds: number        // 마을 총 침대 수, 0이면 숙소 없음
     services: Service[]
   }

   기존 sv 문자열('wbspax')을 Service[] 로 변환하고,
   id는 nameEs를 케밥케이스로 슬러그화해서 생성해줘.

4. data/routes.ts
   프랑스 길 전 구간 / 레온 출발 / 사리아 출발 3개 루트 정의.
   각각 시작 마을 id와 표시명, 설명, 슬러그를 갖는다.

완료 조건:
- npm run dev 가 에러 없이 뜬다
- npm test 가 실행된다 (테스트는 아직 없어도 됨)
- data/towns.ts 에 82개 마을이 있고, km 값이 오름차순으로 정렬되어 있다
- 마지막 마을이 santiago-de-compostela, km 773.1 이다

하지 마:
- 마을 데이터의 거리/고도 값을 임의로 반올림하거나 수정하지 마
- 없는 마을을 추가하지 마
```

## PROMPT 1 — 일정 분할 엔진

```
lib/planner/ 에 구간 분할 엔진을 구현해줘. 순수 함수로만 작성하고,
React나 Next.js에 의존하지 마. 나중에 React Native로 그대로 옮길 코드야.

파일:
  lib/planner/types.ts
  lib/planner/split.ts
  lib/planner/split.test.ts

인터페이스:

export interface PlanInput {
  startTownId: string
  targetKmPerDay: number      // 14 ~ 32
  fitness: 'low' | 'normal' | 'high'
  restDays?: number           // 기본 0
}

export interface Stage {
  dayNo: number
  fromTownId: string
  toTownId: string
  distanceKm: number          // 소수점 1자리
  elevationGain: number       // 양수만, m
  warnings: StageWarning[]
}

export type StageWarning =
  | 'STEEP_CLIMB'        // 오르막 400m 초과
  | 'LONG_DISTANCE'      // 28km 초과
  | 'FEW_BEDS'           // 도착지 침대 30개 미만
  | 'NO_SERVICES'        // 구간 내 물/식당 없는 거리 10km 초과

export interface Plan {
  stages: Stage[]
  totalKm: number
  totalDays: number
  compostelaEligible: boolean   // 100km 이상
  doubleStampRequired: boolean  // 사리아 이후 구간 포함 여부
}

알고리즘 (02_기능사양서.md F-01 참조):

1. 체력 보정: low ×0.85, normal ×1.0, high ×1.15
2. 탐욕 분할: 현재 위치 + 목표거리 지점을 계산하고,
   beds > 0 인 마을 중 그 지점에 가장 가까운 곳을 도착지로 선택. 산티아고까지 반복
3. 후처리:
   a. 마지막 구간이 6km 미만이면 직전 구간에 병합
   b. 오르막 400m 초과 구간은 STEEP_CLIMB 경고
   c. 28km 초과는 LONG_DISTANCE 경고
   d. 도착지 beds < 30 이면 FEW_BEDS 경고
4. 100km 미만이면 compostelaEligible = false

테스트 (반드시 작성):
- 생장 출발 24km/normal → 33~35 구간이 나온다
- 사리아 출발 → 총 거리 114.2km, compostelaEligible = true
- 사리아 출발 → doubleStampRequired = true
- 모든 구간의 fromTownId는 이전 구간의 toTownId와 일치한다
- 모든 구간의 도착지는 beds > 0 이다 (산티아고 제외)
- 마지막 구간의 toTownId는 항상 santiago-de-compostela 다
- 거리 합계가 총 거리와 오차 0.1km 이내로 일치한다
- 목표 14km(최소값)에서도 무한루프에 빠지지 않는다
- 목표 32km(최대값)에서도 모든 도착지에 숙소가 있다

완료 조건:
- npm test 전부 통과
- split.ts 안에 fetch, localStorage, window 참조가 없다
```

## 프롬프트 작성 시 지킨 원칙

이 문서를 직접 수정할 때 참고.

1. **한 프롬프트 = 한 관심사.** 데이터, 로직, UI, SEO를 섞지 않는다
2. **완료 조건을 숫자로.** "잘 동작한다"가 아니라 "82개 마을, km 오름차순"
3. **하지 말 것을 명시.** 에이전트는 안 시킨 일을 하는 쪽으로 기운다
4. **테스트 케이스를 프롬프트에 넣는다.** 나중에 시키면 자기 코드에 맞춘 테스트를 쓴다
5. **도메인 함정을 미리 알려준다.** "공립 알베르게는 예약을 안 받는다" 같은 것
6. **타입 정의를 직접 준다.** 이름이 흔들리면 뒤 단계가 전부 어긋난다
